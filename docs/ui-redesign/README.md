## Аудит

### Сильные стороны проекта
- Продукт уже имеет чёткий operational-фокус: турнир, раунд, заезд, судейская работа, spectator.
- Тёмная среда подходит под использование на площадке и в условиях повышенной концентрации.
- `Spectator` хорошо сформулирован как отдельный публичный экран с большим таймером.
- В системе есть важный для продукта слой hardware/readiness/status, который может стать сильной частью UX.

### Основные проблемы

#### UX-проблемы
- Пользовательский сценарий раздроблен между списком турниров, таблицей заездов, судейской панелью и spectator.
- Серверное состояние и readiness важны, но визуально ощущаются второстепенными.
- `History` присутствует в навигации, но не выглядит как рабочий полезный режим.
- `Next round`, `Replay`, `Play`, `Edit results` и другие действия недостаточно объясняют состояние системы.
- В таблице оператору приходится считывать слишком много мелких действий из одной строки.

#### UI-проблемы
- Иерархия заголовков, таблиц и служебной информации слабая.
- Слишком много длинных пустых тёмных плоскостей и тонких границ.
- Боковая навигация занимает много места при малом количестве пунктов.
- У кнопок нет достаточно ясной и постоянной приоритетности.
- В некоторых частях интерфейса есть следы битой кодировки, что снижает доверие к продукту.

#### Проблемы навигации
- Навигация не отражает реальные режимы работы соревнования.
- Судейская панель отделена маршрутом, хотя воспринимается как продолжение сценария раундов.
- Нет единого центра управления активным раундом.

### Рекомендации
- Перестроить операторский интерфейс вокруг operational workflow, а не только вокруг страниц.
- Сделать более заметной верхнюю статусную зону: сервер, readiness, sensor, активный турнир.
- Снизить визуальную тяжесть навигации и поднять значимость центральной рабочей зоны.
- Перевести работу с заездами из «табличного микроменеджмента» в более ясный сценарный контроль.
- Разделить рабочие действия, публичное отображение и служебные настройки визуально и структурно.

## Концепции

### Концепция A — Эволюционная
Цель:
- сохранить знакомую структуру;
- улучшить читаемость;
- уменьшить визуальный шум;
- минимизировать переобучение операторов.

Подход:
- компактнее сайдбар;
- чище таблицы;
- отдельный статусный top bar;
- contextual panel для выбранного заезда;
- узнаваемая цветовая среда остаётся.

Реализуемость:
- высокая;
- хороший кандидат для поэтапного внедрения.

### Концепция B — Профессиональная
Цель:
- собрать полноценный operations cockpit;
- ускорить работу пользователя;
- сделать интерфейс зрелым и профессиональным.

Подход:
- верхняя command bar;
- две основные рабочие зоны: queue/navigation слева, detail workspace справа;
- судейский режим встроен в общую рабочую модель;
- данные и действия структурированы как система управления событием.

Реализуемость:
- средняя;
- лучший баланс между улучшением UX и внедряемостью.

### Концепция C — Инновационная
Цель:
- предложить смелую альтернативу;
- сделать продукт запоминающимся;
- усилить ощущение control room.

Подход:
- модульные bento-блоки;
- крупные статусные сегменты;
- динамическая сценарная композиция;
- меньше «обычных таблиц», больше operational tiles и lane views.

Реализуемость:
- средняя/высокая;
- хороша как сильный продуктовый рывок, но потребует более смелой перестройки.

## Что рекомендовано
Базовая рекомендация — **Концепция B**.

Почему:
- лучше всего соответствует природе продукта как системы управления соревнованием;
- даёт реальный UX-выигрыш, а не только косметику;
- остаётся технически реалистичной для постепенного внедрения.
