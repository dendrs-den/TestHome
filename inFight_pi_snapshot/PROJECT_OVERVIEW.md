# inflight — Overview

## 1. Что это за система
`inflight` — это система управления соревнованиями по аэротрубным дисциплинам.

По факту это связка из:
- терминала судьи (админский интерфейс),
- экранов для зрителей/табло,
- серверного API-слоя,
- core-сервиса, который управляет состоянием раунда,
- аппаратных сервисов (контроллер пересечения линии, Bluetooth-кнопки, DMX/LED).

Цель системы: вести турнир, запускать/останавливать раунды, фиксировать пересечения стартовой линии, учитывать фолы/скипы, обновлять табло и световые сценарии.

## 2. Основные роли в архитектуре

### 2.1 Frontend (`src_frontend/client`)
Клиентское SPA-приложение (React), в котором есть 3 основных режима:
- `/terminal` — рабочий интерфейс оператора/судьи,
- `/infoboard` — экран для зрителей,
- `/scoreboard` — табло/результаты.

### 2.2 Node/Express сервер (`src_frontend/server`)
Прокси и оркестратор между UI и core:
- поднимает HTTP API (`/tournaments`, `/rounds`, `/actions`, `/bluetooth`, ...),
- отдает собранный frontend,
- делает запросы в core (`CORE_API_URL`, обычно `http://127.0.0.1:15000`),
- через Socket.IO пушит события в UI (старт/стоп раунда, обновление инфо, фолы и т.д.).

### 2.3 Core-сервис (`/opt/inflight/core/cmd/app/main`)
Главный движок предметной логики:
- хранит и ведет состояние раунда,
- обрабатывает команды запуска/остановки/повтора,
- ведет фолы, пересечения, турнирные данные,
- взаимодействует с внешними аппаратными сервисами.

### 2.4 Crossing service (`jsm.crossing-detector-front`)
Сервис аппаратного контроля пересечения стартовой линии.

По конфигу core (`config.json`) есть адрес `crossServiceAddress` (обычно `127.0.0.1:15001`). Это подтверждает, что core опирается на отдельный сервис пересечений.

### 2.5 Bluetooth devices
Поддерживаются BT-устройства (кнопки/пульты):
- получение списка,
- connect/disconnect,
- биндинг кнопки “Next”,
- удаление привязок.

### 2.6 DMX/LED
В core-конфиге есть `DMXaddress` (обычно `127.0.0.1:9090`) и наборы цветов:
- `PreparationColor`,
- `ExpectationColor`,
- `PerformanceColor`,
- `CompletionColor`,
- `FaultColor`.

Это означает, что core управляет световыми сценариями в зависимости от состояния раунда и событий (включая фол).

## 3. Как проходит типичный раунд (упрощенно)
1. Оператор выбирает/активирует нужный раунд.
2. Команда `start` уходит из UI в Node, далее в core.
3. Core переводит систему в состояние выполнения и активирует нужные световые сценарии.
4. Пересечение линии фиксируется через crossing service.
5. Фолы/скипы могут приходить:
- вручную из UI,
- удаленно (в том числе через BT-события/команды).
6. Node отправляет live-события через Socket.IO, UI обновляет таймеры/таблицы/статусы.
7. По `stop` core завершает раунд, формирует данные результата.
8. Результаты отображаются на табло/инфоэкране, система может переключиться на следующий раунд.

## 4. Что хранит и чем управляет система
- Текущий турнир и список турниров.
- Дисциплины, команды, стадии, раунды.
- Состояние раунда (prepare/administration/start/stop).
- История пересечений и фолов.
- Состояние BT-устройств и привязок кнопок.
- Световые состояния (DMX/LED сценарии).

## 5. Сервисная схема на Raspberry Pi
Сейчас на Pi используются systemd-сервисы:
- `inflight-core.service` → `/opt/inflight/core/cmd/app/main`
- `inflight-server.service` → `node /opt/inflight/frontend/server/server.js`
- `inflight-crossfront.service` → `/opt/inflight/bins/jsm.crossing-detector-front`
- `nginx` → reverse proxy на `127.0.0.1:3001`

## 6. Важное замечание по кодовой базе
В проекте есть смешение `TS` и `JS` реализаций. На практике рабочий контур сейчас завязан на JS-роутах (`src_frontend/server/routes/*.js`), а часть TS-контроллеров выглядит как заготовки.

Это важно учитывать при рефакторинге: сначала определить, какие файлы реально участвуют в прод-рантайме, и только потом унифицировать слой API.

## 7. Кратко: что проект должен делать
Проект должен обеспечивать полный цикл проведения соревнования в аэротрубе:
- подготовка и ведение турнирной сетки,
- управление раундами в реальном времени,
- интеграция с датчиком стартовой линии,
- интеграция с BT-кнопками/пультами,
- управление LED-лентами через DMX по состояниям,
- мгновенное отображение статуса и результатов для судей и зрителей.

## 8. Важный Чеклист Для Новой Pi (Чтобы Не Было Вечного Preloader)
На новой установке UI может "висеть" на прелоадере, если backend смотрит не на тот порт core/API.

Проверить обязательно:
- `src_frontend/server/.env`: `CORE_API_URL=http://127.0.0.1:15000`
- в `src_frontend/server/routes/*.js` не должно остаться `127.0.0.1:15010`
- `infight-core-mock` (или реальный core) должен быть запущен и слушать тот же порт, что указан в `CORE_API_URL`

Быстрая проверка:
- `curl http://127.0.0.1:3001/tournaments/getall` должен вернуть JSON (не ECONNREFUSED)
- `systemctl status infight-core-mock infight-web`

## 9. Зафиксированный Прод-Подход (Вариант 2)
Текущий рабочий и подтвержденный путь для продакшена:
- за эталон берем **старую рабочую Pi-систему**, где датчик пересечения уже стабильно работает,
- новую Pi разворачиваем не "с нуля по пакетам", а переносом проверенного runtime-комплекта,
- при невозможности собрать/подключить драйвер под новый kernel используем старый рабочий образ как baseline.

Причина:
- датчик на старой системе работает стабильно,
- на новой системе возможна несовместимость драйвера/модуля с ядром,
- исходники низкоуровневого драйвера могут быть недоступны, поэтому надежнее переносить проверенный бинарный контур.

## 10. Что Должно Входить В Runtime-Пакет Для Чистой Pi
Минимальный обязательный набор для быстрого запуска без ручных "дотягиваний":
- `/opt/inflight/core/cmd/app/main`
- `/opt/inflight/core/config.json`
- `/opt/inflight/bins/jsm.crossing-detector-front`
- `/opt/inflight/bins/jsm.bluetooth` (если используется в текущем сценарии)
- frontend server/client runtime (`/opt/inflight/frontend/...`)
- unit-файлы systemd:
- `/etc/systemd/system/inflight-core.service`
- `/etc/systemd/system/inflight-server.service`
- `/etc/systemd/system/inflight-crossfront.service`
- при необходимости `/etc/systemd/system/inflight-bluetooth.service`
- все требуемые udev/rules и modules-load настройки, связанные с датчиком/контроллером.

## 11. Быстрый Smoke-Test После Установки На Новую Pi
1. Проверить режим запуска:
- `cat /etc/default/infight-mode` (если используется переключатель `MOCK/REAL`)
2. Проверить сервисы:
- `systemctl status inflight-core inflight-server inflight-crossfront`
3. Проверить API:
- `curl http://127.0.0.1:3001/tournaments/getall`
4. Проверить канал датчика:
- `journalctl -u inflight-crossfront -n 100 --no-pager`
- убедиться, что нет постоянных ошибок открытия устройства/порта
5. Проверить реакцию на физическое пересечение:
- при реальном проходе должен появляться event пересечения, без "авто-срабатывания" на старте.

## 12. Зафиксировано В Текущей Сборке
- Режимы `MOCK/REAL` переключаются через `/etc/default/infight-mode`.
- В `REAL` режиме рабочий набор сервисов: `inflight-core`, `inflight-server`, `inflight-crossfront`.
- `infight-web` относится к mock-контуру и в `REAL` может быть отключен.
- Backend унифицирован на `CORE_API_URL` (без жесткой привязки к `127.0.0.1:15010`).
- Добавлен endpoint `GET /utils/getIp` для Infoboard/Spectator потока.
- `/bluetooth/getall` при недоступном bluetooth-runtime возвращает корректный `200` с пустыми массивами (без падения UI).
- Убрана зависимость от legacy `/round/save` в потоке завершения раунда (не засоряет логи 404).
- Для battle-раундов работает операторская логика:
- минимум 2 слота по умолчанию,
- `ADD` добавляет пустой слот,
- `DEL` доступен для пустой строки или строки без результатов.

Если пункт 5 не проходит:
- сначала сверить kernel/драйвер/модуль с рабочей старой Pi,
- затем переносить недостающие бинарники/модули именно из рабочего образа.
