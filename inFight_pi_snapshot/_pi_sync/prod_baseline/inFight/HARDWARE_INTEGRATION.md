# inflight — Hardware Integration

## 1. Назначение документа
Этот документ описывает аппаратную часть `inflight`: как связаны Raspberry Pi, датчик пересечения, Bluetooth-контроллер и DMX-управление LED-лентами.

## 2. Аппаратный контур (верхнеуровнево)

`Raspberry Pi (Core + Node + UI)`
- центральная точка управления,
- принимает команды оператора,
- получает сигналы от аппаратных сервисов,
- управляет логикой раунда и световыми сценариями.

`Crossing detector service`
- подключен к датчику стартовой линии через внешний контроллер,
- фиксирует факт и время пересечения,
- передает событие в core.

`Bluetooth devices`
- беспроводные кнопки/пульты для удаленных действий,
- используются для удаленного запуска/остановки/фолов и навигации по сценарию.

`DMX controller + LED strips`
- принимает команды от core по адресу `DMXaddress`,
- переключает световые сцены (подготовка, ожидание, выполнение, завершение, фол).

## 3. Логическая схема данных и команд
1. Оператор дает команду в UI (terminal) или событие приходит с BT-кнопки.
2. Node API проксирует команду в core (`CORE_API_URL`).
3. Core меняет состояние раунда.
4. Core отправляет:
- команды на DMX (цвет/сцена),
- запросы/ожидание событий от crossing/bluetooth сервисов.
5. Node через Socket.IO рассылает обновления в интерфейсы (`terminal`, `infoboard`, `scoreboard`).

## 4. Ключевые адреса из конфига core
(по `deploy_core/cmd/app/config.json`)
- `DMXaddress`: `127.0.0.1:9090`
- `crossServiceAddress`: `127.0.0.1:15001`
- `bluetoothServiceAddress`: `127.0.0.1:15002`

## 5. Таблица: событие → реакция системы

| Событие | Источник | Что делает core | Что видит UI | Что делает свет (DMX/LED) |
|---|---|---|---|---|
| Переход в подготовку раунда | Оператор/автологика | Ставит state=prepare | Обновление статуса и текущего раунда | `PreparationColor` |
| Переход в ожидание | Core | Ставит state=expectation | Таймер/статус ожидания | `ExpectationColor` |
| Старт раунда | Оператор или remote start | Запускает раунд и таймер | Событие `stateStart`, запуск секундомера | `PerformanceColor` |
| Первое пересечение линии | Датчик через crossing service | Фиксирует crossing, обновляет round-info | Обновление данных пересечений/таймера | Обычно остается `PerformanceColor` |
| Фол/скип | UI или remote fault | Добавляет запись фола, валидирует | Обновление таблицы фолов | Мигание/сцена `FaultColor` на `FaultColorTime` |
| Завершение раунда | Оператор/remote stop | Закрывает раунд, формирует результат | `stateStop`, обновление результата | `CompletionColor` |
| Ошибка/неполный проход | Логика core | Помечает проблемный исход | Backdrop/уведомления в UI | Часто fault/alert-сцена |
| Переход к следующему раунду | Оператор/автологика | `round/set/next` | Событие `nextRound`, обновление таблиц | Возврат к `PreparationColor`/`ExpectationColor` |

## 6. Bluetooth-функции (наблюдаемые API)
Через `/bluetooth/*` маршруты доступны операции:
- `getall` — список устройств,
- `connect/:mac`, `disconnect/:mac`,
- `bind/:guid/:type` — привязка кнопок,
- `dropkeys/:mac`, `dropSetKeys/:mac`,
- `waitfault` — ожидание удаленного фола.

## 7. Что нужно для стабильной эксплуатации на площадке
1. Зафиксированный порядок автозапуска сервисов (`core`, `server`, `crossfront`, `nginx`).
2. Проверка живости датчика и BT перед стартом соревнований.
3. Тест DMX-сцен (prepare/performance/fault/completion) до открытия площадки.
4. Резервный сценарий ручного судейства при потере датчика/BT.
5. Логи и быстрый чек каналов связи (15000/15001/15002/9090).

## 8. Ограничения текущего описания
Этот документ отражает фактическую интеграцию, видимую по коду и конфигам в snapshot.
Для электрических подключений (пины, питание, тип протокола контроллера датчика) нужен отдельный инженерный паспорт оборудования.
